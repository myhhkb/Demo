## 项目简介

本项目是一个辅助英语学习的前后端分离 Web 应用。用户可以通过前端页面查询英文单词，后端系统会调用阿里百炼云 AI 大模型接口生成该单词的中文释义和 3 条例句。查询结果返回前端展示后，用户可以手动点击“保存”按钮，将该单词记录持久化到自己的个人单词本中，便于后续复习。

本项目不仅实现了基础的用户系统与单词管理功能，还重点完成了以下实践目标：

- 使用 **Go + Gin** 搭建 RESTful 后端服务
- 使用 **React + Vite** 完成前端页面开发
- 使用 **JWT** 完成用户认证与接口鉴权
- 使用 **MySQL + GORM** 完成数据持久化与软删除
- 使用 **Docker Compose** 对前后端和数据库进行一键编排部署
- 使用 **Nginx 反向代理** 与 **Vite Proxy** 解决跨域问题
- 接入 **阿里百炼云 AI 接口**，实现智能化单词解释与例句生成

## 项目亮点

1. **AI 查询与持久化解耦**  
   用户查询单词时，系统先调用 AI 接口返回释义与例句；只有在用户点击“保存到单词本”后，系统才会将数据写入数据库，避免无效存储。

2. **用户数据隔离**  
   所有单词数据均与当前登录用户绑定，不同用户之间的单词本彼此隔离。

3. **软删除设计**  
   删除单词时并不直接物理删除数据库记录，而是采用软删除方式，便于后续扩展恢复、统计和审计功能。

4. **前后端分离部署完整**  
   项目同时覆盖开发环境与生产环境的运行方式，具备较完整的工程化实践价值。

5. **跨域处理符合要求**  
   没有在 Go 后端中使用 CORS 中间件，而是分别通过开发环境代理和生产环境反向代理解决跨域。

## 功能展示

### 1. 用户注册与登录

- 用户可通过用户名和密码完成注册
- 登录成功后，后端签发 JWT Token
- 前端保存 Token，用于访问需要鉴权的接口

### 2. AI 智能查询单词

- 用户输入英文单词后发起查询
- 后端调用阿里百炼云 AI 接口
- 支持选择不同模型，如 `qwen` 和 `deepseek`
- 返回内容包括：
  - 单词本身
  - 中文释义
  - 3 条英文例句及中文含义

### 3. 保存到个人单词本

- 查询结果默认只展示，不自动入库
- 用户点击“保存”后，后端将单词记录写入数据库
- 同一用户下相同单词不能重复保存

### 4. 单词本分页列表

- 查看当前登录用户保存过的所有单词
- 支持分页查询，提升列表展示效率

### 5. 删除单词

- 支持删除个人单词本中的记录
- 后端采用软删除实现

## 系统架构说明

本项目采用前后端分离架构，并通过 Docker Compose 编排为三个核心服务：`frontend`、`backend`、`db`。

| 层级 | 服务/组件 | 技术实现 | 主要职责 | 对外暴露 |
|------|-----------|----------|----------|----------|
| 用户访问层 | Browser | 浏览器 | 访问前端页面、发起 API 请求 | - |
| 前端入口层 | frontend | Nginx + React 静态资源 | 提供前端页面，并将 `/api` 请求反向代理到后端 | `80:80` |
| 后端接口层 | backend | Go + Gin | 用户认证、单词查询、单词保存、单词列表、单词删除 | 仅容器网络内部 `8080` |
| 数据持久层 | db | MySQL 8.0 | 保存用户信息和单词记录 | 仅容器网络内部 `3306` |
| 外部服务 | 阿里百炼云 | 通义千问 / DeepSeek API | 生成单词释义和 3 条例句 | 由后端通过 HTTPS 调用 |

### 请求流转过程

1. 用户在浏览器访问 `http://localhost`。
2. `frontend` 容器中的 Nginx 返回 React 打包后的静态页面。
3. 当前端页面请求 `/api/...` 接口时，请求先到达 Nginx。
4. Nginx 通过 `proxy_pass` 将 `/api` 请求转发给 `backend:8080`。
5. 后端完成 JWT 鉴权、业务处理和数据库读写。
6. 当用户查询未保存过的单词时，后端调用阿里百炼云 AI 接口获取释义和例句。
7. 后端将结果返回前端展示；只有用户点击“保存到单词本”时，后端才会写入 MySQL。

### 容器访问关系

| 调用方 | 被调用方 | 访问地址 | 用途 |
|--------|----------|----------|------|
| Browser | frontend | `http://localhost` | 访问前端页面 |
| frontend/Nginx | backend | `http://backend:8080` | 反向代理 API 请求 |
| backend | db | `db:3306` | 连接 MySQL 数据库 |
| backend | 阿里百炼云 | `https://dashscope.aliyuncs.com/...` | 调用 AI 大模型接口 |

## 核心业务流程说明

### 1. 注册 / 登录流程

1. 用户提交用户名和密码
2. 后端校验参数并写入或查询用户表
3. 登录成功后生成 JWT Token
4. 前端保存 Token，后续请求放入 `Authorization` 请求头中

### 2. 查询单词流程

1. 用户输入单词并点击查询
2. 前端调用 `/api/word/query`
3. 后端先检查当前用户是否已经保存过该单词
4. 如果已保存，则直接返回数据库中的内容
5. 如果未保存，则调用 AI 接口生成释义和例句
6. 后端将结果返回前端，但不立即入库

### 3. 保存单词流程

1. 用户在查询结果页面点击“保存”
2. 前端将单词、释义、例句和 AI 来源提交给后端
3. 后端将数据与当前用户 ID 绑定后写入数据库
4. 保存成功后可在“我的单词本”中查看

### 4. 删除单词流程

1. 用户在单词列表中点击删除
2. 前端调用删除接口
3. 后端校验该记录是否属于当前用户
4. 验证通过后执行软删除

## 跨域处理说明

本项目**严格遵循**不在后端 Go 代码中配置任何 CORS 中间件的要求。

### 开发环境跨域处理

开发环境下：

- 前端地址：`http://localhost:5173`
- 后端地址：`http://localhost:8080`

由于端口不同，浏览器会认为这是跨域请求。因此项目在 `vite.config.js` 中配置了 `server.proxy`，将前端发出的 `/api` 请求转发到后端服务，从而避免浏览器直接跨域访问。

### 生产环境跨域处理

生产环境下用户只访问：

```text
http://localhost
```

此时：

- React 静态资源由 Nginx 提供
- `/api` 请求同样先到达 Nginx
- Nginx 再将 `/api` 请求反向代理到 `backend:8080`

这样浏览器始终只与同一个源通信，因此不会产生跨域问题。

### 结论

- **开发环境**：通过 Vite Proxy 解决跨域
- **生产环境**：通过 Nginx 反向代理解决跨域
- **后端代码**：未配置 CORS 中间件

## 数据库设计说明

项目主要包含两张核心数据表：

### 1. `users` 用户表

用于保存用户账号信息：

- 用户 ID
- 用户名（唯一）
- 密码哈希值
- 创建时间 / 更新时间

### 2. `words` 单词表

用于保存用户的单词记录：

- 单词 ID
- 用户 ID
- 单词内容
- 中文释义
- 例句 JSON 字符串
- AI 来源
- 创建时间 / 更新时间
- 删除时间（软删除）

其中，`words` 表通过 `user_id` 与 `users` 表建立一对多关系。并通过 `(user_id, word)` 唯一约束，避免同一用户重复保存同一个单词。

更详细的字段设计请查看：

- `docs/db.md`

其次，我对 **前后端协作方式** 的理解更加具体了。以前更多是单独写接口或页面，这次是把前端、后端、数据库和 AI 服务真正串起来，形成完整业务闭环。尤其是在“查询单词”和“保存单词”分离的设计中，我体会到了业务流程设计的重要性。

第三，我对 **Docker Compose 的实际作用** 有了更直观的认识。以前对容器化理解偏概念化，这次通过把前端、后端、MySQL 放到多个容器中统一编排，我真正理解了服务依赖、网络通信、健康检查和一键启动部署的意义。

第四，我掌握了 **开发环境和生产环境跨域问题的不同解决方式**。通过 Vite Proxy 和 Nginx 反向代理，我理解了为什么很多项目并不直接依赖后端 CORS 配置，而是通过网关或代理层统一处理请求。



### 二、此次作业的难点

这次作业中，我觉得主要有以下几个难点。

第一个难点是 **前后端联调**。虽然前端和后端各自功能不算特别复杂，但真正联调时会涉及请求路径、请求头、Token 传递、响应格式统一等很多细节，只要有一个地方不一致，页面就可能无法正常展示数据。

第二个难点是 **JWT 鉴权流程的实现**。包括登录成功后生成 Token、前端保存 Token、后续接口自动携带 Token，以及后端中间件解析并识别当前用户，这一整套流程需要前后端配合好，否则就会出现未登录、Token 无效、用户数据错乱等问题。

第三个难点是 **AI 接口返回结果的处理**。大模型接口虽然功能强，但返回内容可能不够稳定，因此在提示词设计、结果解析和前端展示上都需要考虑格式统一问题，保证返回内容能稳定用于页面显示和数据库保存。

第四个难点是 **Docker 部署与服务通信**。例如数据库服务需要先启动并健康可用，后端才能连接；前端请求在生产环境中也不是直接访问后端端口，而是要通过 Nginx 转发。这些服务之间的依赖关系和通信地址，刚开始比较容易混淆。

第五个难点是 **跨域问题的理解与处理**。尤其是在开发环境中，浏览器访问的前端端口和后端端口不同，很容易出现请求失败。通过这次作业，我才真正把“为什么会跨域”和“如何通过代理解决跨域”理解清楚。

总体来说，这次作业不仅让我完成了一个功能完整的小项目，也让我对接口开发、工程部署、跨域处理和 AI 能力接入有了更全面的实践认识。
