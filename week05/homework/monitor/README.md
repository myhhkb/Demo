## 项目简介

这是一个使用 Go 实现的命令行服务健康探测工具。程序会从 `config.json` 中读取多个探测目标，自动识别并执行 `HTTP` 或 `TCP` 健康检查，利用 goroutine 并发扫描所有任务，并在全部任务完成后生成一份 CLI 汇总报告和日志文件。

## 已完成功能

- 支持通过 `config.json` 配置多个探测目标。
- 支持 `HTTP` 与 `TCP` 两类探测协议。
- 若配置中未显式填写 `protocol`，可根据地址自动识别协议类型。
- 支持每个任务单独配置 `retry_count`，且限制最大重试次数不超过 3。
- 支持通过 `--timeout` 控制单个任务超时时间。
- 支持通过 `-v` 开启详细模式，实时输出每个目标的探测结果。
- 使用 goroutine 并发执行所有探测任务。
- 每个任务内保留 `time.Sleep(1 * time.Second)`，方便演示并发性能效果。
- 所有任务结束后汇总生成终端报表。
- 程序运行结束后自动生成日志文件，格式为 `monitor-log-YYYYMMDDHHMMSS.log`。
- 提供单元测试，覆盖配置解析、协议识别、重试机制、结果汇总和日志生成。

## 配置文件说明

配置文件格式如下：

```json
{
  "targets": [
    {
      "name": "百度搜索",
      "address": "https://www.baidu.com",
      "protocol": "http",
      "expected_status": 200,
      "retry_count": 1
    }
  ]
}
```

字段说明：

- `name`：服务名称。
- `address`：探测地址，HTTP 使用完整 URL，TCP 使用 `host:port`。
- `protocol`：可选，支持 `http` 和 `tcp`；不填时会自动识别。
- `expected_status`：HTTP 期望状态码，默认 `200`。
- `expected_body`：可选，要求响应体中包含指定内容。
- `retry_count`：可选，失败后重试次数，最大不超过 `3`。

说明：

- `MySQL 服务` 与 `本身 SSH` 属于本地 TCP 检测，若你的电脑没有开启对应端口，失败是正常现象。
- `慢速响应模拟` 默认访问 `https://httpbin.org/delay/5`，如果命令行使用 `--timeout 3`，它通常会超时，从而验证超时控制是否生效。
- `Raw 文本文件` 会检查响应体中是否包含 `Go`。

## 实现思路

1. 先解析命令行参数，读取配置文件。
2. 对每个目标执行配置校验，并自动补全默认值。
3. 使用 goroutine 并发启动所有探测任务。
4. 每个任务内部根据协议分别走 HTTP 或 TCP 探测逻辑。
5. 若任务失败且配置了 `retry_count`，则在同一个 goroutine 中自动重试。
6. 所有结果通过 channel 回收，等待全部 goroutine 执行完成后统一汇总。
7. 生成成功率、响应时间分布、最快服务、最慢服务等统计信息。
8. 输出 CLI 报告，并将相同内容写入日志文件。

## 开发中遇到的问题与解决方案

### 1. 如何同时支持 HTTP 和 TCP 两种探测方式

**问题：**
刚开始我只会写一种网络请求方式，不知道怎么在同一个程序里兼容网页地址和端口地址。

**解决方案：**
我把探测逻辑拆成 `checkHTTP()` 和 `checkTCP()` 两个函数，再通过 `protocol` 字段或地址前缀自动识别该走哪一条分支。这样主流程更清晰，也方便后续扩展更多协议。

### 2. 如何保证所有任务并发执行且最后统一汇总

**问题：**
如果一个个顺序检测，慢接口会拖慢整体完成速度，也不符合题目要求的高并发探测。

**解决方案：**
我使用 goroutine 为每个目标单独启动任务，再通过 `WaitGroup` 等待所有任务结束，最后从 channel 中收集所有结果。这样既能并发执行，也能保证最终报表在全部任务完成后再输出。

### 3. 如何避免单个目标卡住整个程序

**问题：**
像 `delay/5` 这种慢接口或者不可访问域名，可能会导致程序长时间等待。

**解决方案：**
我在每次探测时都使用 `context.WithTimeout` 设置超时控制。这样无论是 HTTP 请求还是 TCP 连接，只要超过设定时间就会自动取消，不会拖住整个任务池。

### 4. 如何实现失败后的有限重试

**问题：**
题目要求 `retry_count` 为可选项，而且最大只能到 3，需要避免无限重试。

**解决方案：**
我在配置加载阶段就校验 `retry_count` 范围；执行时统一通过 `CheckWithRetry()` 控制尝试次数，按“首次执行 + 重试次数”总共运行固定轮次，逻辑比较稳定。

### 5. 如何做并发性能演示

**问题：**
如果网络环境太快，视频演示时不容易直观看出任务是并发执行而不是串行执行。

**解决方案：**
按照题目要求，我在每个探测任务中加入了 `time.Sleep(1 * time.Second)`。这样如果有很多任务，总耗时依然会明显小于串行执行时间，更容易在终端演示并发优势。
