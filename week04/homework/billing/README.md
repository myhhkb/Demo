## 项目功能

- 按照三档阶梯电价计算基础电费。
- 根据峰时与低谷时段对总费率进行调整。
- 程序启动时通过 `init()` 打印计费规则版本号和系统初始化时间。
- 在 `main` 函数中调用单独的控制台函数，引导用户输入用电量和用电时段，并输出账单明细。
- 提供单元测试，覆盖基础阶梯计费、峰谷时段判断和最终费用计算。

## 文件说明

- `go.mod`：Go 模块定义文件。
- `billing.go`：业务核心代码，包含初始化、阶梯计费、峰谷时段调整和控制台交互逻辑。
- `billing_test.go`：单元测试代码。
- `assets/`：用于放置测试通过截图等资料。

## 实现思路

1. 使用常量定义三档阈值、各档电价以及峰谷调节系数，避免魔法数字。
2. 将计费流程拆分为多个函数：
  - `runBillingConsole()`：负责输入输出。
  - `CalculateBaseFee()`：负责基础阶梯计费。
  - `IsPeakHour()`：负责判断是否为高峰时段。
  - `ApplyTimeFactor()`：负责应用峰谷调节。
  - `CalculateFinalFee()`：负责串联完整计费流程。
3. `init()` 在程序启动时输出版本号和初始化时间，满足题目要求。
4. 单元测试按“基础费用 + 时段判断 + 最终费用”三个层次编写，便于验证关键逻辑。

## 函数列表及功能说明

- `init()`：打印计费规则版本号和系统初始化时间。
- `main()`：程序入口，只调用控制台执行函数。

- `IsPeakHour(hour int) bool`：判断时段是否属于高峰时段。

## 跨阶梯计费的逻辑难点

跨阶梯计费的关键难点在于：当总用电量超过某一档阈值后，不能把全部电量都按照新档位单价计算，而是必须把不同区间的电量拆开分别计费。例如 500 度电应拆成：前 200 度按第一档、200 到 400 度按第二档、400 度以上部分按第三档。为避免计算错误，我把每一档的费用分段累加，这样可以清晰表达“分段计价”的规则。

## 开发过程中遇到的问题与解决方案

### 1. 不会写 `TestCalculateFinalFee` 这种测试函数

**问题：**
刚开始我只会写业务函数，不知道 Go 的测试函数应该怎么写，尤其是不清楚为什么函数名必须写成 `TestCalculateFinalFee` 这种形式，也不知道 `t *testing.T` 这个参数有什么用。

**解决方案：**
我先查清楚了 Go 单元测试的基本规则：

- 测试文件名必须以 `_test.go` 结尾
- 测试函数名必须以 `Test` 开头
- 参数必须写成 `t *testing.T`

然后先从最简单的单条测试开始写，确认 `got` 和 `want` 的比较方式，再扩展成多组测试用例。这样就能逐步写出完整的 `TestCalculateFinalFee` 函数。

### 2. 不理解 `t *testing.T`，也不明白指针在这里的作用

**问题：**
在测试函数中看到 `t *testing.T` 时，我一开始不理解 `*` 的含义，也不清楚为什么这里不是普通变量，而是一个指针类型。

**解决方案：**
通过查资料和实际使用，我理解到这里的 `t` 是 Go 测试框架传进来的测试对象，`*testing.T` 表示它是一个指针类型。当前阶段不需要自己去操作底层内存，只需要知道可以通过它调用：

- `t.Run()`：执行子测试
- `t.Fatalf()`：测试失败时输出信息并停止

这样理解之后，我就能正常使用测试对象，而不会被“指针”这个概念卡住。

### 3. 不会写表驱动测试，看不懂 `[]struct{}`

**问题：**
在编写测试时，我第一次接触到这种写法：

- `[]struct { ... }`

刚开始完全看不懂它的含义，也不知道为什么测试数据可以这样组织。

**解决方案：**
后来我理解到，这其实是“匿名结构体切片”，很适合拿来存放一组测试用例。每个测试用例都可以包含：

- `name`：测试名称
- `usage`：输入的用电量
- `hour`：输入的时间
- `want`：预期结果

把多组数据统一放在一个切片里，再配合循环逐个测试，代码就会更清晰，也更容易扩展。

### 4. 不理解 `want` 和 `got` 是什么意思

**问题：**
刚开始写测试时，我不知道 `want` 为什么这样命名，也不明白它和 `got` 分别代表什么。

**解决方案：**
后来我明白了：

- `got` 表示程序实际运行得到的结果
- `want` 表示我预先手工计算出的正确结果

测试本质上就是比较这两个值是否一致。如果 `got != want`，就说明程序逻辑有问题。这种命名方式是 Go 测试里很常见的一种写法。

### 5. 不会写 `for _, tt := range tests`

**问题：**
在遍历测试数据时，我一开始不理解为什么要写：

- `for _, tt := range tests`

特别是不明白 `_` 和 `tt` 的含义。

**解决方案：**
后来我知道了，`range` 遍历切片时会返回两个值：

- 当前元素下标
- 当前元素内容

如果下标不需要用，就用 `_` 忽略掉；`tt` 表示当前这一个测试用例。这样就可以把 `tests` 里的每组数据依次取出来进行测试。

### 6. 不会使用 `t.Run()` 写子测试

**问题：**
虽然我能写出普通的测试，但一开始不会写 `t.Run()`，也不知道为什么测试函数里还能再嵌套一个匿名函数。

**解决方案：**
后来我了解到，`t.Run()` 的作用是给每一组测试数据单独命名并执行。这样当某一组失败时，输出信息会更清楚，方便定位是哪一个 case 出了问题。理解这一点后，我就把多组测试改成了子测试形式。

### 7. 不会使用 `fmt.Scanln(&usage)`，尤其不理解 `&`

**问题：**
在写控制台输入时，我不知道为什么 `Scanln` 必须写成 `fmt.Scanln(&usage)`，而不能直接写 `fmt.Scanln(usage)`。

**解决方案：**
后来我明白了，`Scanln` 需要拿到变量的地址，才能把用户输入的内容写进变量里。`6usage` 中的 `&` 表示“取地址”。如果不传地址，函数就无法修改变量值。所以这里必须写：

- `fmt.Scanln(&usage)`
- `fmt.Scanln(&hour)`

这也是我在这个项目里真正开始理解指针使用场景的一个地方。

### 8. 不会区分什么时候用 `var`，什么时候用 `:=`

**问题：**
写 Go 代码时，我经常分不清应该写：

- `var usage float64`

还是：

- `usage := 0.0`

**解决方案：**
在这个项目里我逐渐理解了它们的区别：

- 需要先声明变量、后续再接收输入时，适合用 `var`
- 可以直接根据右侧表达式推断类型时，适合用 `:=`

所以像 `usage`、`hour` 这种需要先声明再传地址输入的变量，我使用 `var`；像 `finalFee` 这种可以直接根据函数返回值得到类型的变量，我使用 `:=`。



### 9. 一开始把所有逻辑都写进 `main()`，导致后面不好测试

**问题：**
最开始我有过把输入、计费、判断时段、打印结果全部写进 `main()` 的想法，但后来发现这样代码很乱，也不方便写单元测试。

**解决方案：**
后来我把逻辑拆成多个函数：

- `runBillingConsole()`：负责控制台输入输出
- `CalculateBaseFee()`：负责基础阶梯计费
- `IsPeakHour()`：负责判断高峰时段
- `ApplyTimeFactor()`：负责峰谷调节
- `CalculateFinalFee()`：负责组合完整流程

拆分之后，每个函数职责更明确，也能针对核心函数单独编写测试，整个程序结构更清晰。
