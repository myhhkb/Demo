### 2. 浏览器调试

#### 1）是什么

调试是指在一个脚本中找出并修复错误的过程。所有的现代浏览器和大多数其他环境都支持调试工具 —— 开发者工具中的一个令调试更加容易的特殊用户界面。它也可以让我们一步步地跟踪代码以查看当前实际运行情况。开发者工具（DevTools）提供了 Elements、Console、Network、Sources 等面板用于调试。

##### 1.1 常用调试面板

| 面板 | 主要作用 | 本项目中的典型用途 |
| --- | --- | --- |
| Elements | 查看和修改 DOM / CSS | 检查布局、边距、字体是否生效 |
| Console | 查看日志和报错 | 定位运行时错误、打印接口结果 |
| Network | 查看请求和响应 | 检查接口状态码、请求头、返回数据 |
| Sources | 断点调试代码 | 跟踪异步逻辑、查看变量变化 |
| Application | 本地存储管理 | 检查 `localStorage` 中的 token |

##### 1.2 一个基本判断思路

当页面出现问题时，我通常按下面顺序排查：

1. 先看 **Console** 有没有报错；
2. 再看 **Network** 请求是否成功；
3. 最后看 **Elements** 和样式是否被覆盖。

> **提示**：很多“页面没显示”的问题，根因并不在页面本身，而是在接口失败或数据为空。

#### 2）与上下文的关系

它连接了"页面代码"和"运行结果"，是验证 HTML/CSS/JS 是否生效的最终环节。

#### 3）在作业中的使用

看 Network 面板定位接口失败、看 Elements 面板定位布局问题以及看 Console 定位运行时错误。

例如在登录、学生管理、工作台数据加载过程中，都需要通过 Network 面板确认：

```text
请求是否发出 → 状态码是否正确 → 返回数据结构是否符合预期
```

#### 4）遇到的问题以及解决方案

- **图片 404**：发现路径未映射。解决：在 Node.js 中配置静态资源转发。
- **状态更新滞后**：解决：在 Sources 打断点确认异步数据结构。
- **登录失败无提示**：解决：结合 Network 与前端拦截器逻辑，定位到 401 被全局跳转提前拦截的问题。

### 3. HTML

#### 1）是什么

HTML（HyperText Markup Language，超文本标记语言）是一种标记语言，用于告诉浏览器如何组织网页内容。它不是编程语言，而是由一系列元素组成的结构描述语言——你用这些元素来包裹内容的不同部分，使其以特定方式显示或运作。

###### 关键概念

| 概念 | 说明 |
| ---- | ---- |
| 标记语言 | 用标签给文本添加结构和含义，而非执行逻辑运算 |
| 超文本 | 通过链接将文档与文档连接在一起，构成互联的网络 |
| HTML 文档 | 扩展名为 `.html` 的文本文件，通常入口文件命名为 `index.html` |

##### 1.1 HTML 的主要职责

HTML 更关注“内容是什么”，而不是“内容长什么样”。例如：

| 标签 | 语义 | 示例用途 |
| --- | --- | --- |
| `h1~h6` | 标题 | 页面标题、模块标题 |
| `p` | 段落 | 正文说明 |
| `a` | 链接 | 页面跳转 |
| `img` | 图片 | Logo、封面图 |
| `form / input / button` | 表单交互 | 登录、搜索、新增数据 |
| `table` | 表格数据 | 学生列表、课程列表 |

##### 1.2 为什么强调语义化

语义化标签不仅让代码更清晰，还会影响：

- 页面可读性；
- SEO（搜索引擎理解页面内容）；
- 无障碍访问（辅助技术识别页面结构）。

> **提示**：能用语义化标签时，尽量不要全都用 `div` 代替。

#### 2）与上下文的关系

HTML 提供结构，CSS 负责样式，JS 负责行为。三者是前端最基础的组合。

#### 3）在作业中的使用

- `week01` 灵犀项目中使用基础标签搭建聊天界面：

```html
<header id="chatHeader">
    <img src="assets/logo.png">
    <span>灵犀 AI</span>
</header>
```

- 在管理系统中，虽然页面主体由 React JSX 编写，但底层仍然是在组织 HTML 结构，例如标题、表单、表格、按钮等。

```tsx
<div className="flex items-center justify-between mb-6">
  <h1>学生管理</h1>
  <Button>新增学生</Button>
</div>
```

#### 4）遇到的问题以及解决方案

- **表单刷新**：解决：`e.preventDefault()` 阻止默认行为。
- **标签闭合导致布局乱掉**：解决：开启编辑器插件（Prettier）自动纠错。
- **结构语义混乱**：解决：尽量按标题、列表、表格、表单等真实含义组织标签。

### 4. CSS

#### 1）是什么

CSS（Cascading Style Sheets，层叠样式表）是用于控制网页外观的样式语言。HTML 定义内容的结构和含义，CSS 定义内容的呈现方式。

##### 1.1 浏览器默认样式

即使你没有写任何 CSS，浏览器也会给 HTML 应用一套内置的默认样式——标题会比正文大，段落之间有间距，链接带有颜色和下划线。这些默认样式确保页面在没有 CSS 的情况下仍然可读。CSS 的作用就是让你接管这些样式的控制权，按你想要的方式呈现页面。

##### 1.2 CSS 能做什么

| 能力 | 示例 |
| --- | --- |
| 文本样式 | 字体、颜色、大小、行高、对齐 |
| 布局 | Flexbox、Grid、多列布局 |
| 装饰 | 背景、边框、阴影、圆角 |
| 动画 | 过渡效果、关键帧动画 |
| 响应式 | 让同一页面适配手机、平板、桌面 |

##### 1.3 核心理念

> HTML 管“是什么”，CSS 管“长什么样”。

不要用 HTML 控制外观（如 `<font>` 标签），也不要用 CSS 表达内容含义。职责分离是前端开发的基本原则。

##### 1.4 常见样式维度

| 维度 | 常见属性 |
| --- | --- |
| 文本 | `color`、`font-size`、`line-height` |
| 盒模型 | `margin`、`padding`、`border` |
| 布局 | `display`、`position`、`flex` |
| 视觉 | `background`、`box-shadow`、`border-radius` |
| 交互 | `transition`、`:hover`、`transform` |

#### 2）与上下文的关系

有了 HTML 结构后，CSS 让页面"可读、可用、可维护"；进入 React 后，CSS 是组件视觉一致性的保障。

#### 3）在作业中的使用

- 使用 Flex 布局完成基础样式，并实现单行省略：

```tsx
style={{ maxWidth: '240px', overflow: 'hidden', textOverflow: 'ellipsis', whiteSpace: 'nowrap' }}
```

- 登录页中通过渐变背景、卡片阴影、圆角等方式提升视觉层级。
- 学生管理页中通过边框、留白、Tag 和按钮颜色区分不同信息区域。

> **提示**：一个页面“看起来舒服”，很多时候不是因为用了多复杂的技术，而是因为间距、层级、对齐和配色处理得足够稳定。

#### 4）遇到的问题以及解决方案

- **垂直居中难题**：解决：全面改用 Flexbox。
- **样式污染**：解决：使用 CSS Modules 或内联样式限制作用域。
- **文本溢出难看**：解决：通过 `white-space + overflow + text-overflow` 做单行省略处理。

### 5. JavaScript 编程基础

#### 1）是什么

JavaScript 负责页面逻辑与数据处理，包括变量、条件、循环、函数、对象数组操作等。

##### 1.1 JavaScript 主要解决什么问题

| 类型 | 典型能力 |
| --- | --- |
| 交互控制 | 点击、输入、弹窗、表单提交 |
| 数据处理 | 遍历、过滤、映射、合并 |
| 异步行为 | 请求接口、定时器、事件监听 |
| 页面更新 | 根据数据动态渲染内容 |

##### 1.2 基础能力示例

```javascript
function sendMessage() {
  const message = messageInput.value.trim();
  if (!message) return;
  addMessage('user', message);
}
```

上面的逻辑体现了 JS 最典型的用途：**获取输入 → 判断条件 → 调用函数 → 更新页面**。

#### 2）与上下文的关系

它是 Web API、异步请求、框架开发的基础。没有 JS 基础，后续 React 和工程化很难深入。

#### 3）在作业中的使用

- 处理消息发送逻辑，并使用 `map` 渲染列表：

```tsx
{students.map((student) => <tr key={student.id}><td>{student.name}</td></tr>)}
```

- 在学生管理、课程管理等模块中，还经常使用 `filter`、`join`、对象展开等方式组织数据。

#### 4）遇到的问题以及解决方案

- **浅拷贝坑**：解决：使用展开运算符 `{...oldObj}`。
- **this 指向**：解决：统一改用箭头函数。
- **数组处理混乱**：解决：明确区分 `map`（映射）、`filter`（过滤）、`find`（查找）的职责。

> **提示**：JS 基础看似简单，但项目里最常见的 bug 往往就出在变量作用域、数据结构理解和数组操作上。

### 6. Web APIs，DOM 和 BOM

#### 1）是什么

- **DOM**: 操作页面节点与事件。
- **BOM**: 浏览器对象模型（`window.location`、`localStorage` 等）。

##### 1.1 三者关系

| 概念 | 关注对象 | 典型 API |
| --- | --- | --- |
| DOM | 页面文档结构 | `querySelector`、`addEventListener` |
| BOM | 浏览器窗口环境 | `location`、`history`、`localStorage` |
| Web APIs | 浏览器提供的能力集合 | `fetch`、`setTimeout`、`EventSource` |

##### 1.2 为什么它们重要

JavaScript 语言本身只是一套语法，真正让它“能操作网页、能请求接口、能读写本地存储”的，是浏览器提供的这些 API。

#### 2）与上下文的关系

JS 语言本身不直接操作页面，DOM/BOM/Web APIs 提供了与浏览器环境交互的入口。

#### 3）在作业中的使用

- 使用 `localStorage` 维持 Token，使用 `window.location.href` 处理跳转。
- 在灵犀项目中通过事件监听处理用户输入和滚动行为；
- 在管理系统中通过浏览器存储保持登录状态。

```typescript
if (error.response?.status === 401) {
  localStorage.removeItem('token');
  window.location.href = '/login';
}
```

#### 4）遇到的问题以及解决方案

- **Token 过期**：解决：在响应拦截器中捕获 401 并清空存储。
- **事件绑定时机不对**：解决：确保 DOM 已加载完成后再绑定事件，或在框架组件生命周期中处理。
- **本地存储和状态不同步**：解决：把“读取 / 写入 / 清除”逻辑统一放在认证流程中管理。

> **提示**：DOM、BOM、Web APIs 是“JS 走向真实页面”的桥梁，理解它们之后，很多前端现象都会更容易解释。

## 二、JavaScript 语言进阶和工程化

### 1. 异步编程和 Promise

#### 1）是什么

异步编程用于处理非阻塞操作，如网络请求。Promise 是其规范，`async/await` 是其终极语法糖。

##### 1.1 为什么需要异步

在浏览器和 Node.js 中，很多任务都不是“立即完成”的，例如：

| 场景 | 是否适合异步 | 原因 |
| --- | --- | --- |
| 发起 HTTP 请求 | 是 | 需要等待服务端返回 |
| 读取文件 | 是 | 依赖磁盘 IO |
| 定时任务 | 是 | 需要等待时间到达 |
| 复杂纯计算 | 不一定 | 更偏向同步或交给 Web Worker |

> **提示**：异步的核心不是“代码更高级”，而是“不要阻塞主线程”。

##### 1.2 Promise 与 `async/await` 的关系

- `Promise` 解决了回调地狱问题；
- `async/await` 让异步代码看起来更像同步代码；
- `try...catch` 则让异常处理路径更加清晰。

```javascript
async function loadData() {
  try {
    const result = await fetch('/api/data');
    return await result.json();
  } catch (error) {
    console.error('加载失败', error);
  }
}
```

#### 2）与上下文的关系

它是 HTTP 请求、实时通信的基础。React 页面加载、接口错误处理都依赖异步模型。

#### 3）在作业中的使用

API 调用统一使用 `async/await`：

```tsx
const res = await studentsApi.getStudents({ page, ...filters });
```

在学生管理页和登录页中，还会配合 `try...catch`、loading 状态一起使用，让“请求中 / 成功 / 失败”三个状态都可控。

#### 4）遇到的问题以及解决方案

- **并发请求性能**：多个互不依赖的请求按顺序 await 导致加载慢。解决方案：改用 `Promise.all([p1, p2])` 并发执行，大幅缩短总耗时。
- **错误被吞掉**：只写了 `await` 没有统一兜底。解决方案：关键逻辑统一放入 `try...catch` 中，并在页面给出提示信息。

### 2. JS 对象、原型、设计模式

#### 1）是什么

- **原型/继承**: JS 的属性共享机制。
- **类 (Class)**: 封装数据和方法的语法。
- **设计模式**: 针对特定问题的通用方案（如单例、观察者）。

##### 1.1 核心理解

JavaScript 虽然语法灵活，但在项目中仍然需要清晰的数据结构和职责划分。对象负责描述数据，原型负责共享能力，设计模式负责组织代码。

| 概念 | 作用 | 在项目中的体现 |
| --- | --- | --- |
| 对象 | 描述业务数据 | `student`、`course` |
| 原型/类 | 组织复用能力 | JS/TS 基础认知 |
| 设计模式 | 提高可维护性 | API 封装、统一响应工具 |

#### 2）与上下文的关系

从"写代码"到"写架构"的跨越。工程化阶段对代码组织要求更高。

#### 3）在作业中的使用

- 后端 `success/fail` 响应工具函数的封装体现了统一输出的设计。

```javascript
export function success(ctx, data = null, msg = 'success') {
  ctx.body = { code: 0, msg, data };
}
```

> **提示**：这里虽然没有刻意写复杂设计模式，但“统一封装、统一出口”本身就是一种工程化思维。

#### 4）遇到的问题以及解决方案

- **重复代码太多**：每个接口都写同样的错误处理。解决方案：提取公共拦截逻辑到 API 基类或拦截器中。

### 3. 从 XMLHttpRequest 到 Fetch、SSE、WebSocket

#### 1）是什么

- **Fetch**: 现代 Promise 风格请求。
- **SSE (Server-Sent Events)**: 服务端向客户端单向推送文本流。
- **WebSocket**: 全双工长连接交互。

##### 1.1 三种通信方式对比

| 技术 | 通信方向 | 典型场景 | 特点 |
| --- | --- | --- | --- |
| Fetch | 请求-响应 | 获取列表、提交表单 | 简单直接 |
| SSE | 服务端单向推送 | AI 输出流、通知流 | 实现轻量 |
| WebSocket | 双向实时通信 | 聊天室、协同编辑 | 交互能力最强 |

#### 2）与上下文的关系

这是通信能力升级的路径：从"一问一答"到"流式持续更新"。

#### 3）在作业中的使用

- 在 `week02` SSE 练习中发送实时消息：

```javascript
res.writeHead(200, { 'Content-Type': 'text/event-stream' });
res.write(`data: ${JSON.stringify({ text: 'chunk' })}\n\n`);
```

前端则通过 `EventSource` 持续接收服务端返回的片段数据，并逐步更新页面内容。

#### 4）遇到的问题以及解决方案

- **SSE 自动重连问题**：网络抖动导致连接断开。解决方案：利用 `EventSource` 自带的重连机制，并在后端配置合适的 `retry` 间隔。
- **流式数据格式错误**：换行符不规范会导致前端无法正确识别。解决方案：严格按 `data: ...\n\n` 的格式输出。

### 4. Node.js

#### 1）是什么

Node.js 让 JS 运行在服务端，具备访问文件系统、数据库和网络的能力。

#### 2）与上下文的关系

Node.js 串联了全栈链路。通过写 Node.js，更深刻理解了 HTTP 请求的生命周期。

#### 3）在作业中的使用

`week03` 后端基于 Koa 框架，实现了 JWT 验证和数据库 CRUD：

```javascript
ctx.state.user = jwt.verify(token, JWT_SECRET);
```

#### 4）遇到的问题以及解决方案

- **中间件顺序问题**：权限验证放在了路由后面导致失效。解决方案：调整 `app.use` 顺序，确保 `authenticateToken` 中间件在业务路由之前执行。

### 5. Webpack 和 Vite

#### 1）是什么

- **Webpack**: 强大的打包器，适合复杂构建。
- **Vite**: 基于浏览器原生 ESM 的构建工具，开发效率极高。

##### 1.1 二者对比

| 工具 | 特点 | 适用场景 |
| --- | --- | --- |
| Webpack | 生态成熟、配置灵活 | 复杂历史项目 |
| Vite | 启动快、热更新快 | 中小型现代前端项目 |

#### 2）与上下文的关系

解决模块化开发到浏览器部署的转换。Vite 极速的热更新彻底改变了开发体验。

#### 3）在作业中的使用

- `week02` 练习 Webpack 配置，`week03` 全面改用 Vite 提升开发体验。

```ts
const api: AxiosInstance = axios.create({
  baseURL: '/api',
  timeout: 10000,
});
```

上面配合 Vite 的代理配置，可以让前端开发环境下直接请求 `/api`。

#### 4）遇到的问题以及解决方案

- **环境变量读取问题**：Vite 默认不读取 process.env。解决方案：使用 `import.meta.env` 并配置 `.env` 文件前缀为 `VITE_`。

### 6. TypeScript

#### 1）是什么

TS 是 JS 的超集，为变量提供显式类型定义，由微软开发维护。

##### 1.1 TypeScript 的价值

| 价值 | 说明 |
| --- | --- |
| 类型提示 | 写代码时更容易发现错误 |
| 接口约束 | 前后端字段更统一 |
| 可维护性 | 大项目更容易协作 |
| 自文档化 | 通过类型就能理解数据结构 |

#### 2）与上下文的关系

在大中型项目中，TS 是"文档"也是"护城河"，显著减少因字段错写导致的低级 Bug。

#### 3）在作业中的使用

定义 `Student`, `Course` 等 interface 约束 API 返回值：

```typescript
export interface Student {
  id: number;
  name: string;
  status: 'active' | 'inactive';
}
```

> **提示**：当接口字段越来越多时，TypeScript 的价值会迅速放大，因为它能把“运行时报错”提前成“编码时报错”。

#### 4）遇到的问题以及解决方案

- **后端返回数据类型不匹配**：API 定义的 interface 与后端实际返回不符。解决方案：使用工具类 `ApiResponse<T>` 封装，并严格校验后端数据字典。

## 三、HTTP 入门和 React 开发

### 1. HTTP 原理与实践

#### 1）是什么

HTTP 是互联网通信基石。定义了语义化的方法（GET/POST 等）和标准状态码。

##### 1.1 常见方法与含义

| 方法 | 含义 | 本项目示例 |
| --- | --- | --- |
| GET | 获取资源 | 获取学生列表 |
| POST | 创建资源 | 新增学生 |
| PUT | 更新资源 | 编辑学生信息 |
| DELETE | 删除资源 | 删除学生 |

##### 1.2 常见状态码

| 状态码 | 含义 |
| --- | --- |
| 200 | 请求成功 |
| 201 | 创建成功 |
| 400 | 参数错误 |
| 401 | 未认证/认证失效 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

#### 2）与上下文的关系

它是前端向后端索取数据的"通用语言"。

#### 3）在作业中的使用

- `week03` 接口遵循 RESTful 风格，创建成功返回 201 状态码。

```javascript
ctx.status = 201;
success(ctx, student);
```

#### 4）遇到的问题以及解决方案

- **跨域问题 (CORS)**：本地测试连不上后端。解决方案：在 Vite 中配置 Proxy，通过本地同源请求转发绕过浏览器同源策略限制。

### 2. Whistle 和 Postman 使用

#### 1）是什么

- **Postman**: 接口调试神器。
- **Whistle**: 抓包代理工具。

#### 2）与上下文的关系

它们是调试 HTTP 的"显微镜"。

#### 3）在作业中的使用

- 在前端代码还没写好时，先用 Postman 测通后端路由。
- 在联调过程中，可借助 Whistle 或浏览器 Network 面板确认请求头、返回体和状态码。

#### 4）遇到的问题以及解决方案

- **Token 携带失败**：解决：通过 Whistle 抓包确认 Header 是否存在拼写错误。

> **提示**：调接口时不要只看“有没有报错”，还要看“请求参数是否真的按预期发出去了”。

### 4. React 技术栈

#### 1）是什么

React 是声明式、组件化的 UI 库，通过 Virtual DOM 提升性能。

##### 1.1 React 的核心思维

- UI = state 的映射
- 组件是页面最小复用单元
- 页面更新不是“手动操作 DOM”，而是“修改状态后自动重渲染”

##### 1.2 常用 Hook

| Hook | 作用 |
| --- | --- |
| `useState` | 管理局部状态 |
| `useEffect` | 处理副作用/请求 |
| `useMemo` | 缓存计算结果 |
| `useCallback` | 缓存函数引用 |

#### 2）与上下文的关系

让页面逻辑从"操作 DOM"变成"管理数据"。

#### 3）在作业中的使用

- Dashboard、课程管理等页面全部由 React 组件和 Hooks 构成：

```tsx
const [students, setStudents] = useState<Student[]>([]);
useEffect(() => { loadStudents(); }, [page, filters]);
```

此外，登录页、工作台、课程页、学生页都按照页面组件拆分，使职责更清晰。

#### 4）遇到的问题以及解决方案

- **无限循环渲染**：在 useEffect 里修改了依赖项。解决方案：严格检查依赖数组，或使用函数式更新 `setCount(prev => prev + 1)`。

### 5. React Router

#### 1）是什么

React 官方推荐的路由方案，实现 SPA 的页面切换。

##### 1.1 路由的价值

| 能力 | 说明 |
| --- | --- |
| 页面切换 | 不刷新页面切换视图 |
| 路由守卫 | 根据登录状态控制访问 |
| 嵌套路由 | 管理后台布局复用 |
| 编程式跳转 | 通过代码控制页面跳转 |

#### 2）与上下文的关系

实现页面逻辑的物理隔离与跳转。

#### 3）在作业中的使用

- 实现登录受限制路由保护与重定向：

```tsx
<Routes>
  <Route path="/login" element={user ? <Navigate to="/" /> : <LoginPage />} />
  <Route path="/*" element={user ? <LayoutPage user={user} /> : <Navigate to="/login" />} />
</Routes>
```

#### 4）遇到的问题以及解决方案

- **路由无法匹配**：解决：检查 `/*` 通配符的使用，确保子路由能正确分发。

### 6. Zustand

#### 1）是什么

Zustand 是一个轻量级、响应式的状态管理库，常用于替代复杂的 Redux。

##### 1.1 它适合什么场景

- 多个页面共享同一份用户信息；
- 不想层层传递 props；
- 不希望引入 Redux 那样较重的样板代码。

#### 2）与上下文的关系

解决"组件层级太深导致 props 传递困难"的问题。

#### 3）在作业中的实践

- 学习了通过 Store 集中管理用户全局状态，避免 Props 层层传递的问题。

#### 4）遇到的问题以及解决方案

- **状态持久化**：刷新页面后 Store 数据丢失。解决方案：结合 `persist` 中间件将状态同步至 LocalStorage。
